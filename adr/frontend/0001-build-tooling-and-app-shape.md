# 0001. ビルド基盤とアプリ形態: Vite + SPA の維持

- ステータス: Accepted
- 日付: 2026-06-14

## コンテキスト

フロントエンド(`frontend/`)は以前に作られたまま構成が古びている、という前提で
見直しを始めた。まず土台である「ビルドツール」と「アプリの形態(SPA / SSR / SSG)」を
決めないと、後続の設計(ルーティング・データ取得・認証)が定まらない。

調査の結果、依存パッケージ自体は新しかった(React 19 / Vite 7 / Tailwind v4 /
TypeScript 5.8 / React Router 7 / shadcn-ui)。問題はバージョンではなく設計とコード品質にある。
したがってビルド基盤を入れ替える積極的な理由はない。

アプリ形態については、SSR/SSG の主な利点(初回表示の速さ・SEO・OGP)が
このアプリに当てはまるかが論点になる。本アプリは **ログイン必須の private な管理画面**であり、
検索流入や SNS シェアの対象ではない。

## 決定

### 1. ビルドツールは Vite を維持する

Vite は 2026 時点でも第一線であり、現状の構成も最新に近い。乗り換えコストに見合う利益がないため、
**Vite を据え置く**。

### 2. アプリ形態は SPA を維持する

ログイン必須の private アプリのため、SSR/SSG の主な利点は効かない。

- SEO / OGP は不要(認証の背後にあるため、そもそもクローラ・シェアの対象外)。
- 静的ファイルをそのまま配信でき、デプロイが単純(Node サーバ不要)。
- バックエンドの Go API は JSON を返すだけで、フロントと綺麗に分離できる。

SSR(Next.js 等)を採らない理由は **コストではなく**、上記の利点が private app に効かないことと、
Go API と BFF 層の二重管理を避けるため。コストを度外視しても判断は変わらない。

### 3. SPA の弱点は後続 ADR で対策する

SPA に残る弱点は、それぞれ別の決定で塞ぐ。

| SPA の弱点 | 対策する ADR |
|---|---|
| 初回表示が遅い(JS を全部落とすまで描画されない) | コード分割・ルート遅延ロード([ADR-0002](0002-routing-react-router-framework-mode.md) / [ADR-0005](0005-directory-structure-feature-based.md)) |
| トークン置き場の XSS リスク | 認証([ADR-0004](0004-auth-and-token-management.md)) |

## 結果

### 良い点

- 既存資産(Vite 設定・依存)を活かし、移行リスクを最小化できる。
- 静的配信でデプロイが単純なまま、後続の改善に集中できる。

### トレードオフ

- SPA 固有の弱点(初回表示・トークン保存)は残るため、後続 ADR での対策が前提になる。
- 将来 SEO が必要な公開ページが生まれた場合は、本 ADR の再検討が要る。
