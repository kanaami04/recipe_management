# convention-review — コードを第三者視点で規約・既存コードの慣習と突き合わせる

**どこを直すと「このリポジトリの書き方」に揃うか** を file/line で挙げる定性レビュー。正確性バグ・見た目・lint/typecheck/formatter が拾うもの(整形・import 漏れ・型エラー)は対象外。

サービス／ロジック層・新規モジュール・構造や依存方向の変更(フレームワーク移行やコード生成系の差し替えを含む)など、設計・規約に効く差分を push 前に確かめたいときに回す。瑣末な差分(ドキュメント・整形・定常的な依存バンプ)は対象外。実装本人の自己確認は甘くなるので、回すと決めたら第三者を通す。

## このリポジトリの前提

- **正は「既存コードに倣う」。** このプロジェクトは方針を ADR で明文化せず、既存の実装と git 履歴をお手本にする。従うべきことの多くは条項でなく既存コードに宿る。
- **明文規約は CLAUDE.md に集約。** ルート [CLAUDE.md](../../../CLAUDE.md) と領域別 [api/CLAUDE.md](../../../api/CLAUDE.md) / [frontend/CLAUDE.md](../../../frontend/CLAUDE.md)。Claude Code は対象ファイルの所在に応じて領域別を自動で読み込む。

## 照合は 2 層で見る

条項違反(明文)と既存の書き方からの逸脱(暗黙)は別物。**両方**を別々に見る。

| 層 | 何を正にするか | 担当する差分 |
| --- | --- | --- |
| **第 1 層: 明文規約照合** | CLAUDE.md(ルート + 領域別)の条項 | 条項違反。総当たりで決定的に拾える |
| **第 2 層: 既存慣習レビュー** | 同領域の既存実装 | 条項に無い崩れ＝「このリポジトリではこう書く」からの逸脱 |

第 2 層の方が広い。このリポジトリは「既存コードに倣う／方針を明文化しない」が基調のため。

## 手順

### 1. 材料をそろえる

- **対象の diff**(base との差分・変更行に絞る。既存の問題は見ない)。
- **効く明文規約をファイルパスで集める**(中身は写さず参照)。ルート [CLAUDE.md](../../../CLAUDE.md)、領域別 CLAUDE.md、触れたファイルのコメント・lint ignore。
- **同領域の既存実装**を読み、粒度・命名・配置・依存方向・エラーの扱いの慣習を控える(第 2 層の正)。バックエンドは同じ層(handler/service/repository/domain)の隣接ファイル、フロントは同じ feature の隣接ファイルを見る。

### 2. 第 1 層 — 明文規約を総当たりで照合する

集めた条項を上から順にすべて潰す。条項ごとに「diff の領域に効くか → 効くなら適合しているか」を判定し、効かない条項は「該当なし」と明示する。違反は条項テキストと diff の該当箇所を並べて出す(例: `api/CLAUDE.md「リポジトリ抽象は domain に置く」 vs service/foo.go:42 で GORM 直呼び … 違反`)。オラクル(事前に渡された指摘)があっても総当たりの代わりにせず、最後の答え合わせに使う。

### 3. 第 2 層 — 既存の書き方への準拠を第三者に見させる

同領域の既存実装に照らし、条項に無い崩れを見る。Agent ツールで別コンテキストのサブエージェントを **Opus 系(`claude-opus-4-8` 以降)を明示して** 起動する(規約・既存パターンへの適合判断＝高度な推論が主タスクのため)。実装意図・弁護は渡さず、diff・同領域の既存実装・第 1 層の違反リストを grounding として渡す。プロンプトに必ず含める:

> あなたは実装会話を持たない第三者のレビュアーです。diff・同領域の既存実装・明文規約と照合済みの違反リストを渡します。基準は一般論の設計の良し悪しでなく、このリポジトリの既存実装です。これを踏まえ、(1) 違反リストに**載っていない**崩れ、(2) 同領域の既存実装の書き方(粒度・命名・配置・依存方向・エラーの扱い)に倣えているか、を見て良い点と直したい点を挙げてください。各指摘は **どのファイルの何行目・どの関数や型か** を特定し、根拠が「規約」か「既存慣習」のどちらか、「直すべき差」か「意図的な逸脱」かを明示してください。正確性バグと見た目は見ません。好みレベルの差や linter が拾う整形、具体的な直し方は挙げず、観察に徹してください。

### 4. トリアージして反復する

第 1 層・第 2 層の指摘を等しく仕分けし、残差が次のいずれかになるまで手順 2〜3 を回す。

- **既定は「直すべき差」。** 直して再照合する。
- **「意図的な逸脱」は理由を記録する。** PR 本文に逸脱ごとに 1 行(file/line / 内容 / 理由)。理由はスコープ外・段階的移行の途中・コメントで宣言済みなどに限り、自己判定で素通しさせない。
- **規約自体への疑問・規約と既存コードの矛盾は、勝手に直さず確認へ回す。** このリポジトリは「必要になったら都度変える」方針。規約が現状に合っていなければ、違反として黙って直すのでなく CLAUDE.md の更新をユーザーに提案する。

## やってはいけないこと

- **規約を発明しない。** 「規約違反」と言うなら条項テキストと出典(CLAUDE.md の該当行)を示す。明文に無いものは「既存慣習」と明示し、条項違反と区別する。
- **自分好みの設計に引き寄せない。** 基準は一般論の良し悪しでなく既存実装。確立したパターンから外れた点だけを指摘し、一度きりの書き方や好みの差は挙げない。
- **バグと見た目を混ぜない。** 正確性バグは `/code-review`、UI の見た目は Playwright スクショ(frontend/e2e)で見る。本スキルは規約・慣習だけを見る。
