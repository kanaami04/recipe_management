# readable-docs — 読みやすい文書を書く

読み手が最小の労力で必要なものに辿り着ける文書にするための文章作法を扱う。文書の真価は文章量ではなく読みやすさ。読みにくい文書は読み手の労力とコンテキストを浪費し、メンテナンスコストを上げ、寿命を縮める。

## 位置づけ — 構造ではなく文章を扱う

- Issue / PR の**構造**(どのセクションを置くか)は、そのリポジトリの既存 Issue / PR の慣習に従う。本スキルはそれを前提に補完する。
- 本スキルが扱うのは、**各セクションの中身をどう書くか**(文章レベルの読みやすさ)。

## 原則

### 要点を先に出す
セクションの冒頭で「何の話か・結論・読み手が次にすること」が分かるようにする。読み手は冒頭だけで把握できるとよい。経緯や背景は、必要な読み手が読み進めれば得られる位置に置く。

### 1 文 = 1 アイデア
1 つの文に複数の話題を入れない。長い文は読み手に文脈の再構成を求める。2 つのことを言う文は 2 文に分ける。

### 重要な情報を括弧に埋めない
軽い補足を括弧に入れるのは問題ない。ただし、読み手が見落とすと困る**重要な情報**を括弧に押し込まない。括弧の中は流し読みで飛ばされやすい。重要なら独立した文に昇格させる。

### スキャンできる構造にする
見出し・箇条書き・表を使い、拾い読みで目的の箇所に辿り着けるようにする。読み手は最初から最後まで読むとは限らない。説明に段落が必要なら、その段落をリストや表で置き換えられないか検討する。

### 削る
オッカムの剃刀を適用する。「消して何が壊れるか」を確かめ、何も壊れないなら消す。情報を足すより、ノイズを削る。

## よくある失敗と直し方

具体的な失敗例(before / after)は [ANTIPATTERNS.md](ANTIPATTERNS.md) を参照。

## 組版

- 和文と半角英数の間に半角スペースを入れる(例: "Issue を 3 件"、"`gh` を使う")。
- `**強調**` を多用しすぎない。
- 箇条書きは要点を簡潔に。動作確認・TODO は `- [ ]`。
- 関連・関係は、図表で示せるなら図表にする。
