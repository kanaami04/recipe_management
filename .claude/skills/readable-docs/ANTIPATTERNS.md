# よくある失敗と直し方

readable-docs の原則に反する典型例を、before / after で示す。

## 1. 詳細・冗長すぎる

要点が情報の山に埋もれ、読み手が「結局どうすればよいか」を自分で探すことになる。要点を先に出し、深い詳細は別文書に分けてリンクする。

before:
> このバグは認証ミドルウェアがトークンの有効期限を検証する際に境界条件の扱いに問題があり、具体的には期限ちょうどのトークンが…(10 行続く)

after:
> **症状**: 期限ちょうどのトークンが拒否される。
> **原因**: 期限チェックが `<` で境界を含んでいない。
> **修正**: `<=` にして解決できる。

## 2. 文章が長い

1 文が複数の仕事を抱えると、読み手は文脈を組み立て直すことになる。1 文 1 アイデアに分ける。

before:
> この変更はフロントエンドのコンポーネントを更新しつつ、同時にバックエンドの API 仕様も変えるため、デプロイ順序に注意が必要で、また既存のテストも書き換える必要がある。

after:
> この変更はフロントエンドと API 仕様の両方に触れる。そのためデプロイ順序に注意する。既存テストの書き換えも必要である。

## 3. 重要な情報を括弧に埋める

読み手が見落とすと困る情報を括弧に押し込むと、流し読みで飛ばされる。重要なら独立した文に昇格させる。軽い補足の括弧は残してよい。

before:
> この API は非推奨(後継は `/v2` だが移行期限は未定で、旧クライアントは対応が必要になる)。

after:
> この API は非推奨。後継は `/v2`。移行期限は未定で、旧クライアントは要対応。

## 4. 1 つの文書に詰め込みすぎる

背景・設計・実装手順・テスト計画・議論ログを 1 つの Issue に全部入れると、どれも読みにくくなる。1 文書 1 目的に分け、リンクで繋ぐ。

- Issue 本体は「背景 / 問題」と「ゴール」に絞る。
- 実装の記録は PR 本文に置く。
- Issue / PR 本文は単一の真実に保つ。コメントを積み上げず本文を差し替える。
