## ADR（設計判断の参照）

作業を始める前に必ず [adr/README.md](adr/README.md) を読むこと。これが ADR の索引で、各 ADR へのリンクと運用ルールが書かれている。

- README の「一覧」にある ADR は**すべて目を通し、その決定に従って**実装・レビューする。
- ADR はレイヤーごとに分かれている。バックエンド(`api/`)の作業時は [adr/api/](adr/api/) の ADR、フロントエンド(`frontend/`)の作業時は [adr/frontend/](adr/frontend/) の ADR を参照する。
- 新しい設計判断をしたときは、README の命名規則・フォーマットに従って該当ディレクトリへ ADR を追加する。

## 動作確認・ビジュアル確認

UI の変更（特にレイアウト・モバイル対応・見た目）をしたときは、できるだけ実際に動かして確認すること。次の 3 つを状況に応じて使う。

- **`/run`** … アプリを起動してブラウザで開き、スクリーンショットを撮る。「動いてるところを見せて」「この画面のスクショ」系に使う。
- **`/verify`** … 変更が意図どおり動くかをアプリを起動して操作・観察して確認する。「この修正が効いているか確かめて」系に使う。
- **Playwright**（`frontend/` に同梱） … 任意のビューポート幅（例: 375px のスマホ、768px のタブレット）でページを開き `screenshot()` で撮る。モバイル対応の確認に最適。API はバックエンド不要のルートモックで差し替えられる（[frontend/e2e/recipes.spec.ts](frontend/e2e/recipes.spec.ts) の `mockApi` を参照）。
